Module Overview

Module completed (Oct‚ÄìNov 2025) Group Project + Individual Project ePortfolio hosted on GitHub Pages

This ePortfolio captures my learning journey through the Secure Systems Architecture module. The module builds from core operating-system and distributed-systems concepts to secure design of cloud-native applications and resilient architectures. It connects theory with practice by asking students to design and evaluate an end-to-end system, justify architectural choices, and reflect on their professional growth.

Across the six units I moved from revisiting OS fundamentals such as processes, isolation and access control to reasoning about microservices, APIs, data flows, and failure modes in distributed systems. A central theme of the module is that security is not a final add-on but a structural property shaped by patterns, dependencies, and trade-offs. The group project and individual project both required me to apply these ideas in a realistic scenario and explain my decisions clearly for non-technical stakeholders.

Unit 1: Introduction to Secure Systems Architecture

Lecturecast Reading Formative activity

Focus

Unit 1 introduced the idea of systems architecture as a set of design decisions about components, boundaries, communication paths, and controls. We revisited basic concepts such as layers, trust zones, interfaces, and threat surfaces and linked them to security principles like least privilege, defence in depth, and secure defaults.

Intended Learning

  • Describe what a systems architecture is and why it matters for security and resilience.
  • Explain how trust boundaries, tiers, and interfaces shape the attack surface.
  • Identify early design choices that can improve or weaken security later.

e-Portfolio Artefacts

  • Initial notes mapping a simple three-tier web application and its trust boundaries.
  • Short written reflection on how design decisions made at architecture level influence everyday security tasks.
  • Sketch of a basic layered model (client, edge, application, data) annotated with security controls.

Unit 2: Operating Systems, Isolation and Access Control

Lecturecast Seminar activity

Focus

This unit focused on how operating systems provide isolation and resource control. We looked at processes, threads, memory protection, file permissions and scheduling, then related these concepts to containers and virtual machines. The unit linked OS features to higher level architecture concerns such as multi-tenancy and noisy neighbours in shared environments.

Intended Learning

  • Explain how processes, threads and virtual memory contribute to isolation.
  • Describe how access control mechanisms (users, groups, ACLs) are used to enforce policy.
  • Relate OS-level abstractions to container and virtualisation technologies used in modern architectures.

e-Portfolio Artefacts

  • Short comparison table of processes vs threads and containers vs virtual machines.
  • Notes from the seminar on how poor process isolation or misconfigured permissions can lead to privilege escalation.
  • Mini exercise mapping OS controls (for example file permissions) to architecture decisions (for example log storage, secrets handling).

Unit 4: Distributed Systems and Cloud Architecture Patterns

Reading Pattern exercises Formative work

Focus

Unit 4 deepened the focus on distributed systems and cloud architecture patterns. We discussed concepts like latency, consistency, replication and partitioning and connected them to patterns such as gateways, microservices, queues and event-driven designs. Security considerations included how identity, secrets and policies are enforced across many small services instead of a single monolith.

Intended Learning

  • Describe key trade-offs in distributed systems such as consistency versus availability.
  • Recognise common cloud patterns and how they affect security and observability.
  • Relate design patterns to threat modelling and control placement across services.

e-Portfolio Artefacts (Unit 4)

  • Pattern-mapping notes linking selected patterns from the core text to the module project scenario.
  • Revised architecture sketch incorporating message queues or APIs for clearer separation of concerns.
  • Short written commentary on how pattern choice can reduce or increase attack surface.

Unit 5: Resilience, Monitoring and Observability

Seminar (recording) Formative activity Support for Individual Project

Focus

This unit linked architecture design to resilience and observability. We explored patterns such as timeouts, retries, bulkheads and circuit breakers and discussed how logging, metrics and traces provide feedback about system health. The unit acted as a bridge between high-level design and the more detailed reasoning needed for the individual project.

Intended Learning

  • Explain how resilience patterns protect systems from cascading failures.
  • Outline what effective monitoring and observability look like for a distributed system.
  • Connect resilience and observability to risk management and service level objectives.

Seminar Notes

The seminar reinforced that a secure architecture must also be observable. We looked at how failures, slow dependencies and configuration issues show up in metrics or logs and how these signals link back to design choices. There was also discussion of how to justify architecture decisions in written assignments by referring to specific patterns and trade-offs instead of only describing the final diagram.

e-Portfolio Artefacts (Unit 5)

  • Notes on resilience patterns and how they would apply to the module project scenario.
  • List of suggested metrics and logs for critical services in the architecture.
  • Outline of how to argue for resilience and monitoring design choices in the individual project report.

Reflective Summary (Units 1‚Äì6)

At the start of this module I mostly thought about security in terms of individual controls such as firewalls, permissions or encryption settings. By the end I understood that these controls only make sense inside a clear architecture that defines what the system does, where trust boundaries sit and how components interact.

The early units refreshed core systems topics and made it easier to reason about isolation, resource management and access control. The group project forced me to step back and think at diagram level, which helped when discussing design with other people instead of jumping straight into configuration details. Later units added more depth around distributed systems, patterns and resilience, so I could see why certain designs are easier to observe, scale and secure.

The individual project and reflection gave me space to put everything together. I had to justify why the architecture should look a certain way, acknowledge trade-offs and show how it would be operated and monitored in reality. Going forward I want to:

  1. Keep practising drawing clear architecture diagrams with explicit trust boundaries.
  2. Explain security decisions in terms of patterns and trade-offs instead of only listing controls.
  3. Link architecture decisions to resilience, monitoring and risk management rather than treating them separately.
  4. Continue improving my technical writing so that architecture documents are readable for both technical and non-technical audiences.

Overall, Secure Systems Architecture has made me more confident about discussing design, not just implementation, and has given me a clearer mental model for how secure, resilient systems fit together.
